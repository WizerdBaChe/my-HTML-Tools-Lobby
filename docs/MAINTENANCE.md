# 維護與更新原則 (Maintenance Guide)

本文件用來定義本 repo 的維護方式、更新原則、命名規範與擴充邏輯，避免工具數量增加後出現結構混亂、首頁失控、路徑不一致等問題。

## 維護目標

本 repo 的維護重點不是做成大型前端框架專案，而是持續維持一個清楚、可擴充、低耦合的 HTML 工具集合站。

維護時應優先確保：

- 工具可直接開啟。
- 路徑穩定。
- 首頁可正常載入 `tools.json`。
- 不同工具彼此獨立，不互相污染。
- 使用者可以快速理解每個工具的用途。

## 核心原則

### 1. 工具獨立

每個工具原則上應位於：

```text
tools/<tool-id>/index.html
```

若工具較複雜，可在該資料夾內再加入：

- `style.css`
- `script.js`
- `assets/`

但工具相關資源應優先放在工具自己的資料夾內，不要濫用全域共用資源。

### 2. Lobby 與工具分離

`index.html` 的角色是大廳頁，不應承載具體工具邏輯。

Lobby 只負責：

- 讀取 `tools.json`
- 產生卡片
- 進行基本篩選、分類、搜尋
- 進行 iframe 預覽

不要把某個工具的特殊邏輯硬寫進 Lobby。

### 3. tools.json 是單一資料來源

首頁工具卡片應以 `tools.json` 作為唯一資料來源。

新增、下架、重新分類、標記狀態時，應優先修改 `tools.json`，而不是直接手改首頁卡片 HTML。

### 4. 路徑優先使用相對路徑

所有站內連結、腳本、樣式與工具入口，優先使用相對路徑。

例如：

- `./tools/calculator/`
- `./assets/css/main.css`

避免直接寫死完整 GitHub Pages 網址，以降低 repo 名稱、網域或部署位置變更時的修改成本。

### 5. 先做 MVP，再做擴充

新增功能時，優先順序如下：

1. 先確認功能是否真的需要。
2. 先做最小可用版。
3. 確認不會破壞現有結構。
4. 再考慮視覺美化與互動升級。

避免為了少量需求引入過重框架或複雜建置工具。

## 命名規範

### 工具資料夾命名

工具資料夾統一使用：

- 小寫英文
- kebab-case 命名

例如：

- `json-viewer`
- `image-resizer`
- `text-diff`

避免：

- 空白
- 中文資料夾名
- 大寫混用
- 不可預測縮寫

### tools.json 的 id 規範

`id` 應與工具資料夾名稱一致或高度對應，方便查找與維護。

例如：

```json
{
  "id": "json-viewer",
  "path": "./tools/json-viewer/"
}
```

## 新增工具流程

新增工具時，依以下順序進行：

1. 建立工具資料夾與入口頁：
   `tools/<tool-id>/index.html`
2. 補上該工具所需的 `style.css`、`script.js`。
3. 在 `tools.json` 新增工具描述。
4. 確認 `path` 可正常開啟。
5. 確認是否適合 `iframe` 預覽。
6. 從 Lobby 測試：
   - 卡片是否顯示
   - 開啟是否正常
   - 預覽是否正常
7. 若有顯著更新，補到 changelog 頁。

## 允許與不建議的變更

### 允許

- 新增獨立工具。
- 補充分類、標籤、描述。
- 改善 Lobby 視覺設計。
- 增加搜尋、排序、篩選。
- 新增 about、changelog、help 等靜態頁。

### 不建議

- 把所有工具混寫在首頁。
- 讓不同工具共用過多全域變數。
- 未經整理就把外部第三方程式碼直接丟進 repo。
- 在沒有必要時導入大型框架。
- 破壞既有工具路徑而不做相容處理。

## iframe 使用原則

不是所有工具都適合 iframe。

適合 iframe 的工具通常具備：

- 單頁操作。
- 視窗需求不大。
- 不依賴特殊跳轉流程。
- 不需要過多彈窗或跨頁互動。

若工具不適合內嵌，應將 `supportsIframe` 設為 `false`，並只提供外部開啟。

## 更新原則

### 小更新

以下情況可視為小更新：

- 修正文案。
- 調整樣式。
- 修正工具描述。
- 修補單一工具 bug。

小更新可直接提交，但仍應檢查首頁是否正常載入。

### 中更新

以下情況屬於中更新：

- 新增工具。
- 調整工具分類。
- 增加搜尋或篩選功能。
- 修改 `tools.json` 結構。

中更新應至少進行首頁與工具入口測試。

### 大更新

以下情況屬於大更新：

- 更改整體目錄結構。
- 導入建置流程。
- 改為前端框架架構。
- 大幅改動首頁載入機制。

大更新前應先討論結構是否仍符合本 repo 的原始定位：靜態、清楚、低維護成本。

## 版本與變更紀錄

若修改內容會影響使用方式、路徑或工具列表，應同步更新：

- `pages/changelog/`
- 或另外建立 `CHANGELOG.md`

## 文件維護

當以下內容有變化時，文件也要同步更新：

- 專案結構
- 新增工具流程
- tools.json 欄位規格
- 授權方式
- 部署方式

若程式已更新但文件未更新，應視為不完整更新。

## 授權與引用

若 repo 使用 MIT License 或其他授權，新增第三方程式碼或素材時，應先確認其授權是否相容。

不要把來源不明或禁止再散布的內容直接加入 repo。

## 維護判斷準則

做任何變更前，先問三個問題：

1. 這個功能是否讓工具大廳更清楚，而不是更混亂？
2. 這個改動是否會增加未來維護成本？
3. 這個改動是否真的需要現在做？

只要其中兩題答案偏負面，就應先暫緩。
